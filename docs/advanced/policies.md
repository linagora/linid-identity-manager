# Policies Configuration (OPA)

## 🎯 Purpose

LinId uses **OPA scripts internally** to control how roles and permissions are assigned to users per application.

> Users do **not** need to install or manage OPA themselves.
> All policy management happens inside **LinId UI**, which updates the internal OPA server automatically.

---

## 🧩 Core Concepts

* **Policy**: A script that defines how roles are assigned to users based on attributes, organizational units, or other
  conditions.
* **Active Policy per Application**: Each application has **one active OPA script** that governs role assignments.
* **Rule**: A single, self-contained fragment of a policy. An application policy is built from the concatenation of all
  its **active** rules (rules that are not disabled).
* **Script Editor**: LinId provides a built-in editor to write, edit, and test OPA scripts in **Rego syntax**.

---

## 🐍 A short introduction to Rego

[Rego](https://www.openpolicyagent.org/docs/latest/policy-language/) is the declarative policy language used by OPA
(Open Policy Agent). Instead of writing imperative code (`if / else / return`), you **declare rules** that OPA evaluates
against an `input` document to produce a result.

Key ideas:

* **`input`** — the JSON document OPA receives for each evaluation. In LinId, it describes the user requesting access
  (attributes, organizational units, etc.).
* **Rules** — named declarations that become `true` (or produce a value) when their body holds. A rule body is a list of
  conditions that must **all** be satisfied (logical AND).
* **Sets** — collections that rules **contribute** to with `contains`. Several rules can add to the same set; OPA merges
  the contributions. This is how LinId aggregates the decisions of independent rules.
* **Immutability** — a value assigned with `:=` cannot be reassigned. This is the single most important constraint when
  writing LinId rules (see the fragment contract below).

A minimal rule looks like this:

```rego
# "admin" is added to the roles set when the user is flagged as an administrator.
roles contains "admin" if {
    input.user.admin == true
}
```

You do **not** need to master Rego to write LinId rules — you only need to follow the **fragment contract** below.

## 📐 Fragment contract

Each rule's `script` field must contain a **self-contained Rego fragment**. Fragments of the same application are
concatenated (in ascending `priority` order) into a single policy document. Because they share the same namespace, every
fragment must follow this contract so that fragments can be combined in any order without conflicting.

A fragment **must**:

* **Only contribute** to shared state through partial set rules (`contains`) — never reassign a shared variable with
  `:=`. Rego does not support mutable variables, so `allow := true` in two fragments (or in the template) causes a
  compile error.
* Contribute to the two shared sets owned by the template:
  * **`signals`** — the access decision signals (see below).
  * **`roles`** — the set of roles granted to the user.
* Use **only** the recognized signal values, interpreted by the fixed resolution block generated by the template:
  * **`"allow"`** — grants access if no stronger signal overrides it.
  * **`"force_allow"`** — grants access regardless of other rules, unless overridden by `force_disallow`.
  * **`"force_disallow"`** — always denies access; takes precedence over everything else.
* Evaluate **independently** — a fragment must not assume any execution order relative to the others and must rely only
  on `input`.

A fragment **must not**:

* Redefine the Rego `package` declaration.
* Redefine any template-owned rule (`final_allow`, `result`) or reassign the `signals` / `roles` sets with `:=`.

### Signal precedence

The template resolves the final decision from the `signals` set with the following precedence:

1. `force_disallow` present → **access denied**, whatever else is present.
2. Otherwise `force_allow` present → **access granted**.
3. Otherwise `allow` present → **access granted**.
4. Otherwise → **access denied** (default).

The `roles` set is always returned, independently of the access decision.

### Fragment examples

Grant a role and a simple `allow` signal to members of an organizational unit:

```rego
roles contains "reader" if {
    input.user.organizationalUnit == "public"
}

signals contains "allow" if {
    input.user.organizationalUnit == "public"
}
```

Unconditionally grant access to administrators (`force_allow`):

```rego
roles contains "admin" if {
    input.user.admin == true
}

signals contains "force_allow" if {
    input.user.admin == true
}
```

Always deny access to suspended users (`force_disallow`, wins over everything):

```rego
signals contains "force_disallow" if {
    input.user.suspended == true
}
```

## 🧪 Expected Script Output

The generated policy exposes a single `result` object:

```json
{
  "allow": true,
  "roles": ["admin", "reader"]
}
```

* **`allow`** — a boolean computed from the `signals` set according to the precedence rules above.
* **`roles`** — the deduplicated set of roles contributed by all active fragments.

When no fragment contributes any signal, `allow` defaults to `false` and `roles` is empty.

## ⚙️ How LinId generates and deploys policies

LinId manages the full lifecycle automatically:

1. **Generation** — whenever a rule is created, updated, or deleted, LinId regenerates the application policy from the
   concatenation of its active rules, using a Jinjava template stored on the API classpath.
2. **Change detection** — the generated script and its checksum are stored on the application
   (`applications.script` and `applications.script_checksum`).
3. **Deployment reset** — any regeneration resets the application deployment status (`applications.deployed_at` set to
   `null`), marking it as requiring (re)deployment.
4. **Scheduled deployment** — a background job runs at a configurable interval (`opa.deployment.interval`, default
   `5m`). It pushes every application that has a script but no deployment date to the OPA server, then records the
   deployment timestamp. Deployment failures are logged and retried on the next execution.

The OPA server URL is configured through the `OPA_SERVER_URL` environment variable
(`opa.deployment.server-url`, default `http://opa:8181`). The policy template location is configurable through
`opa.script.location` (default `classpath:templates/opa/opa-policy.rego.j2`).

Each application's policy is isolated in its own Rego package, declared with the bracketed form
`package authz["<application.code>"]` so that codes containing characters such as a dash remain parsable by OPA.

## 🎛️ Customization

The policy generation is designed to be adapted to each client's needs:

* **Template** — LinId ships a **default policy template** (`opa.script.location`, default
  `classpath:templates/opa/opa-policy.rego.j2`). It can be overridden to point to a custom template that better fits a
  client's authorization model, while still receiving the same `application` and `fragments` variables.
* **Input definition** — the structure of the `input` document evaluated by the policies (the description of the user
  requesting access) will be **customizable in a future release**, so clients can shape it to match their own attributes.
* **Result** — the policy result is defined by default as `{ "allow": <bool>, "roles": <set> }`. This contract can be
  **customized** (by adapting the template) to expose additional or different fields required by a client.

> As long as the fragment contract and the shared `signals` / `roles` sets are respected, rule authors do not need to
> know which template variant is in use.

---

## 🔄 Policy Lifecycle

1. **Draft**: Admin writes or edits a rule fragment
2. **Test**: Validate the fragment against test users
3. **Activate**: Enabling a rule includes it in the generated policy; LinId deploys the policy to the internal OPA engine
4. **Update**: Modify and retest when rules change

---

## 📝 Best Practices

* Keep fragments **focused per rule** — one concern per rule.
* Contribute only through `contains`; never reassign `signals`, `roles`, `allow` or `result`.
* Test fragments with sample `input` users before enabling them.
* Document rules and intended behavior for auditing.
* Prefer several small rules over one complex fragment.
* Use `force_disallow` sparingly — it overrides every other rule.
